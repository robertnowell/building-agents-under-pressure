§01Process

The build order, in sequence.

From Anthropic's five-ring tutorial & the writing-tools-for-agents blog. Follow the order.

  1. Read the problem fully. ~30 sec.
  2. Read the starter end-to-end (~90 sec). Inventory what's there and what's missing.
  3. Run it once. Don't fix yet.
  4. State the plan (1–2 min). Architecture + 2–4 tools + workflow vs. agent.
  5. Build minimal end-to-end. One tool, simplest path.
  6. Iterate on tool descriptions + system prompt before adding more tools.
  7. Extend. Tools, edge cases, expansions.
  8. Add evals or document them. "In prod I'd add 5–10 verifiable test prompts with regression alerts."
  9. Wrap up. Name the gap — what you stubbed, what you'd harden.
Start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when simpler solutions fall short. — Schluntz & Zhang · Building Effective Agents
§02Axes

The five axes that matter.

What separates working from polished. Track independently.

Shipping
  • Runs end-to-end
  • Test query returns the right answer
  • Stubbed where the real thing isn't the point
  • Used the time budget
Safety
  • Read-only by default
  • Preconditions on mutations
  • is_error wired through dispatcher
  • Idempotent mutations
  • Refusal paths under uncertainty
Craft
  • Tool descriptions: 3–4 sentences, when-NOT-to-use, examples
  • Real system prompt: role + workflow + format
  • max_turns guard
  • Structured returns (dict, not tuple)
  • Schema props match function params
Judgment
  • Workflow-vs-agent, named early
  • Simplest thing first
  • Tradeoffs surfaced
  • Prod-vs-demo distinctions explicit
Communication
  • Loud thinking — comments, commits, logs
  • Why-not-what on every fix
  • Tradeoffs documented inline
  • Future-you can pick up cold
§03Practice

The discipline behind each axis.

Each axis has a default move. Make it automatic.

Safety // HIGH-LEVERAGE

  • Look at mutating tools first. Add precondition check + idempotency.
Defense lives in the tool, not the prompt. For DB: read-only at the layer (?mode=ro), not just a Python check.
Prod: issue_refund / notify / cancel idempotent on order_id / message_hash / customer_id.

Craft

  • Tool descriptions: 3–4 sentences. When-NOT-to-use. Schema embedded for DB tools. Examples in param descriptions.
  • Returns: list[dict] not list[tuple]. Semantic IDs > UUIDs.
Differentiate tools by WHEN to use them, not only WHAT they do. — Anthropic Troubleshooting

Polish-pass scaffold:

MODEL = "claude-sonnet-4-6"
SYSTEM_PROMPT = "..."

def call_tool(name, tool_input) -> tuple[str, bool]:
    fn = TOOL_FUNCTIONS.get(name)
    if fn is None: return f"unknown tool: {name}", True
    try:    return fn(**tool_input), False
    except Exception as e: return f"{type(e).__name__}: {e}", True

def run_agent(query: str, max_turns: int = 10) -> str:
    messages = [{"role": "user", "content": query}]
    for turn in range(max_turns):
        response = client.messages.create(model=MODEL, max_tokens=4096,
                                          system=SYSTEM_PROMPT, tools=tools, messages=messages)
        if response.stop_reason == "end_turn":
            return next((b.text for b in response.content if b.type == "text"), "")
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    content, is_error = call_tool(block.name, block.input)
                    tool_results.append({"type": "tool_result", "tool_use_id": block.id,
                                         "content": str(content), "is_error": is_error})
            messages.append({"role": "user", "content": tool_results})
            continue
        return f"[unexpected stop_reason: {response.stop_reason}]"
    return "[max_turns reached]"

if __name__ == "__main__":
    print(run_agent("..."))

Judgment

  • Name the architectural choice early.
This is workflow-shaped vs. agent-shaped — let me build the simpler thing first.
In prod I'd narrow the surface to typed intent tools — get_revenue_by_country(country) — never raw SQL. Trade flexibility for safety.

Skip-with-reasoning beats apply-without-thinking.

Shipping

  • Minimal end-to-end first.
  • Test after each tool.
  • Stub aggressively. Mutations return {"id": "REF-XXX", "status": "issued"}.

Communication

  • Inventory the starter first. The inventory is the plan.
  • Every fix: 1–2 sentences of why. "Doing X because [pattern / production reason]."
  • Stuck >20s: read the last turn.
Schluntz & Zhang: start simple, add complexity only when simpler solutions fall short.
§04Gotchas

Gotchas.

The ones you keep hitting.

Python · runtime

# DICT vs OBJECT ACCESS
block.input["title"]              # ✓  block is SDK obj (dot), input is dict (brackets)
block.input.title                 # ✗  AttributeError
# Mental model: SDK metadata = dot. JSON payload = brackets.

# PRINTING SDK OBJECTS
print(f"x: {x}")                  # ✓  always works (uses __repr__)
json.dumps(response.content)      # ✗  TypeError on Pydantic blocks
[b.model_dump() for b in blocks]  # ✓  if you need real JSON

# NEXT() WITHOUT FALLBACK
next((b.text for b in content if b.type == "text"), "")  # ✓ default value
next(b.text for b in content if b.type == "text")        # ✗ StopIteration

# IS_ERROR SEMANTICS — DO NOT INVERT
return result, False              # success
return error_msg, True            # error
# Sanity-check by saying it: "success data, is_error=False"

# KWARGS UNPACKING — SCHEMA PROPS MUST MATCH FUNCTION PARAMS
schema: {"properties": {"query": {"type": "string"}}}
fn(**block.input)                 # → fn(query="...")
def fn(query: str)                # ✓ matches
def fn(expr: str)                 # ✗ TypeError

# TUPLE vs DICT RETURNS FROM TOOLS
return [(start, end), ...]        # ✗ model guesses positions
return [{"start": s, "end": e}]   # ✓ named fields

# WHILE vs FOR(RANGE) FOR LOOPS
while response.stop_reason == "tool_use":  # ✗ no max_turns
for turn in range(max_turns):              # ✓ bounded

# STRING COMPARISON ON TIMES — only safe with zero-padding!
"11:00" < "2:00"   # → True  (because '1' < '2' lexicographically — WRONG)
# Use minutes-since-midnight: int(h)*60 + int(m), THEN compare.

Tool definitions · schema

# PROPERTY TYPES — JSONSchema strings, not Python types
"string"  "integer"  "number"  "boolean"  "array"  "object"  "null"
# integer = ints only · number = float-or-int

# REQUIRED — top-level array; names must match `properties` keys
"required": ["query", "limit"]
# omit it and the model may skip any param without complaint

# ENUM — constrain a string to a fixed set
{"type": "string", "enum": ["pending", "approved", "rejected"]}

# ARRAYS — `items` is mandatory
{"type": "array", "items": {"type": "string"}}

# EMPTY-INPUT TOOLS — still need type=object
"input_schema": {"type": "object", "properties": {}}

# TOOL NAMES — ^[a-zA-Z0-9_-]{1,64}$  (letters, digits, underscore, hyphen)
"query_database"     "query-database"     "issue_refund_v2"

# DESCRIPTIONS — examples in-line beat format hints
{"type": "string", "description": "ISO 8601 date, e.g. '2026-04-28'"}

# DEFAULTS — informational only; apply them in your dispatcher
{"limit": {"type": "integer", "default": 10}}
# limit = inputs.get("limit", 10)   ← actual enforcement lives here

# SCHEMA PROPS MUST MATCH FUNCTION PARAMS — see Python · runtime above
# schema "query" → fn(query=...) · mismatch → TypeError on **inputs unpack

Stdlib · memorize the call

time.time()                            # → float seconds since epoch
from datetime import datetime, timedelta
datetime.now() - last_t                # → timedelta
hash((message, severity))              # → int (in-process dedup OK)
json.dumps(obj, default=str)           # → fallback for non-serializable
re.search(r"\d+", text)                # → match obj or None
§05Trace+Eval

See the loop. Score the loop.

Two helpers. The loop's state, legible at a glance.

Logging — one line per block

Print in two places: after messages.create() and after each dispatch.

# Loop-agnostic. Works for `for turn in range(N)`, `while stop_reason == "tool_use"`,
# recursion, streaming, anything. The `── stop=... ──` line is the turn boundary.

def log_response(response):
    print(f"\n── stop={response.stop_reason} ──")
    for block in response.content:
        if block.type == "text":
            print(f"  text  · {block.text[:120]}")
        elif block.type == "thinking":
            print(f"  think · {block.thinking[:120]}")
        elif block.type == "tool_use":
            print(f"  →  {block.name}({block.input})")

def log_result(name, result, is_error):
    glyph = "✗" if is_error else "✓"
    print(f"  {glyph}  {name}: {str(result)[:120]}")

Wire-in:

# wherever you call the model …
response = client.messages.create(...)
log_response(response)                              # ← here

# … and wherever you dispatch a tool
result, is_error = call_tool(name, tool_input)
log_result(name, result, is_error)                  # ← and here

Want turn numbers? Prepend at the call site.

  • One line per block — glyph column lines up.
  • Truncate to ~120 chars — SQL or row dumps destroy the trace.
  • Glyphs over labels — → call, ✓/✗ result.
  • Always print stop_reason — biggest signal in the loop.
  • Don't dump messages — trace change, not state.

Eval — 4 cases, substring match

Run before declaring "done." Forces you to write down what "working" means.

TESTS = [
    ("How many customers are from France?",       "2"),
    ("Total revenue from France-based customers?", "1575"),
    ("Which product has the most orders?",         "Laptop"),
    ("What is 25 + 30 * 2?",                       "85"),
]

def run_eval(tests):
    passed = 0
    for q, expected in tests:
        try:
            answer = run_agent(q)
            ok = expected.lower() in str(answer).lower()
        except Exception as e:
            answer, ok = f"ERROR: {e}", False
        glyph = "✓" if ok else "✗"
        print(f"{glyph} {q}")
        print(f"   expected: {expected}")
        print(f"   got:      {str(answer)[:120]}\n")
        passed += ok
    print(f"── {passed}/{len(tests)} passed ──")

Smoke test only. Real eval: 5–10 verifiable prompts, tool-call counts and tokens per case, LLM-as-judge calibrated on the first 20 — Writing Tools for Agents.

§06Output

Get structured output reliably.

Schema as tool. Include with your other tools. Model calls it last. Read block.input — already a dict.

return_tool = {
    "name": "return_final_answer",
    "description": "Return the final answer as a structured object. "
                   "Call this exactly once, AFTER you've gathered everything you need.",
    "input_schema": {
        "type": "object",
        "properties": {
            "answer":     {"type": "string", "description": "The final answer."},
            "confidence": {"type": "number", "description": "0.0–1.0."},
            "sources":    {"type": "array",  "items": {"type": "string"}},
        },
        "required": ["answer", "confidence"],
    },
}

# In your loop: include return_tool in `tools=[...]`. The model will call it
# when finished. End the loop the moment you see it.

for block in response.content:
    if block.type == "tool_use" and block.name == "return_final_answer":
        result = block.input     # {"answer": ..., "confidence": 0.92, "sources": [...]}
        return result

Gotchas:

  • Description does the work. "Call this exactly once when done" tells the model when. Reinforce in system prompt.
  • required is enforced. If a field matters, list it.
  • Shape, not semantics. confidence may be the right type and out-of-range. Validate values you care about.
  • Treat it like any other tool. Don't special-case unhappy paths.
§07Reference

Quick reference.

stop_reason values

"end_turn"                        → done; return text
"tool_use"                        → execute tools, append, continue
"max_tokens"                      → truncated; re-issue with higher max_tokens
"pause_turn"                      → server tool iteration cap; re-send as-is
"stop_sequence"                   → rare; treat as end_turn
"refusal"                         → safety; return refusal text
"model_context_window_exceeded"   → terminal; return error

tool_result block — required shape

{
    "type": "tool_result",
    "tool_use_id": block.id,   # MUST match the tool_use id
    "content": str(result),    # or list of {type:"text", text:...}
    "is_error": is_error,      # bool — wire through dispatcher
}

tool_choice options

"auto"                                  # model decides (default)
{"type": "any"}                         # MUST call some tool, model picks which
{"type": "tool", "name": "X"}           # MUST call tool X this turn
{"type": "none"}                        # cannot call any tool
# Forcing is rare. If you force, do it on the LAST turn only —
# mid-loop kills data-gathering.

Troubleshooting · symptom → fix

SymptomFix
Wrong tool selectedSharpen descriptions: WHEN, not WHAT
Tool never calledCheck duplicate names; add input_examples
Wrong param typesAdd "strict": True or input_examples
Invented paramAdd "strict": True
Sequential when parallel expectedSend all tool_results in one user message
Empty response after tool_resultDon't add text block right after; append a new user message
max_tokens mid-tool_useRetry with higher max_tokens; don't submit partial
disable_parallel_tool_use ignoredSet on the request that returns tool_use, not later
