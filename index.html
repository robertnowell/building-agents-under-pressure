§01Process

The build order, in sequence.

From Anthropic's five-ring tutorial & the writing-tools-for-agents blog. Follow the order; resist the urge to leap ahead.

  1. Read the problem fully. Don't skim. ~30 sec.
  2. Read the starter aloud (~90 sec). Narrate what's there and what's missing. Free signal across four axes.
  3. Run it once. See the trace. Don't fix yet.
  4. State plan aloud (1–2 min). Architecture choice + 2–4 tools + workflow vs. agent.
  5. Build minimal end-to-end first. One tool, simplest path. Run.
  6. Iterate on tool descriptions + system prompt before adding more tools.
  7. Extend. More tools, edge cases, expansions.
  8. Add evals or mention them. "In prod I'd add 5–10 verifiable test prompts with regression alerts."
  9. Discuss. Lead with what you'd do in prod that you're not doing here.
Start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when simpler solutions fall short. — Schluntz & Zhang · Building Effective Agents
§02Axes

The five axes that matter.

What separates a working agent from a polished one. Track these independently as you build.

Shipping
  • Code runs end-to-end
  • Test query returns the right answer
  • Stubbed where building the real thing isn't the point
  • Used the time budget; didn't run over
Safety
  • Read-only by default
  • Preconditions on mutating actions
  • is_error wired through the dispatcher
  • Idempotency on mutating actions
  • Refusal paths under uncertainty
Craft
  • Tool descriptions ACI-quality (3–4 sentences, when-NOT-to-use, examples)
  • Real system prompt — named role + workflow + format
  • max_turns safety guard
  • Structured returns (dict not tuple)
  • Schema constraints honored — props match function params
Judgment
  • Workflow-vs-agent named explicitly at minute 3–8
  • Parsimony — simplest thing first
  • Tradeoffs named aloud, not silently chosen
  • Prod-vs-demo distinctions made explicit
Communication
  • Continuous narration; sub-30-second silences
  • Read starter aloud at minute 0–3
  • Why-not-just-what for every fix
  • Engaged in discussion with prepared takes
§03Demo

How to demonstrate each axis.

Each axis has a deploy line. Memorize the rhythm; improvise the words.

Safety // HIGH-LEVERAGE

  • Look at mutating tools first. Add precondition check + idempotency.
Defense should live in the tool itself, not the prompt — prompts are soft guards. For DB tools: read-only at the DB layer (?mode=ro), not just a Python check.
In prod, issue_refund / notify / cancel should be idempotent on order_id / message_hash / customer_id to prevent double-action from loop retries.

Craft

  • Tool descriptions: 3–4 sentences, explicit when-NOT-to-use, schema embedded for DB tools, examples in param descriptions.
  • Returns: list[dict] not list[tuple]. Semantic IDs > UUIDs.
Differentiate tools by WHEN to use them, not only WHAT they do. — Anthropic Troubleshooting

Polish-pass scaffold — type these in the first 90 seconds, every time:

MODEL = "claude-sonnet-4-6"
SYSTEM_PROMPT = "..."

def call_tool(name, tool_input) -> tuple[str, bool]:
    fn = TOOL_FUNCTIONS.get(name)
    if fn is None: return f"unknown tool: {name}", True
    try:    return fn(**tool_input), False
    except Exception as e: return f"{type(e).__name__}: {e}", True

def run_agent(query: str, max_turns: int = 10) -> str:
    messages = [{"role": "user", "content": query}]
    for turn in range(max_turns):
        response = client.messages.create(model=MODEL, max_tokens=4096,
                                          system=SYSTEM_PROMPT, tools=tools, messages=messages)
        if response.stop_reason == "end_turn":
            return next((b.text for b in response.content if b.type == "text"), "")
        if response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    content, is_error = call_tool(block.name, block.input)
                    tool_results.append({"type": "tool_result", "tool_use_id": block.id,
                                         "content": str(content), "is_error": is_error})
            messages.append({"role": "user", "content": tool_results})
            continue
        return f"[unexpected stop_reason: {response.stop_reason}]"
    return "[max_turns reached]"

if __name__ == "__main__":
    print(run_agent("..."))

Judgment

  • Minute 3–8: name the architectural choice.
This is workflow-shaped vs. agent-shaped — let me build the simpler thing first.
In prod I'd narrow the surface to typed intent tools — get_revenue_by_country(country) — never let the LLM write raw SQL. The trade is flexibility for safety.

When briefed on a tactic, name when you'd skip it. Skip-with-reasoning scores higher than apply-without-thinking.

Shipping

  • Build minimal end-to-end first. Don't optimize before the loop runs.
  • Test after each tool added — not four tools then one run.
  • Stub aggressively. Mutating actions return {"id": "REF-XXX", "status": "issued"}. Pretend it worked.

Communication

  • Read starter aloud at minute 0–3. Free signal across four axes.
  • For every fix: 1–2 sentences of why. "I'm doing X because [pattern from anthropic docs / production reason]."
  • When stuck >20s: "Let me read what the agent sent in the last turn."
As Schluntz and Zhang put it — start simple, add complexity only when simpler solutions fall short. That's the heuristic I live by at Kopi.
§04Gotchas

Python traps that actually trip people up.

Not encyclopedic. The ones you keep hitting.

# DICT vs OBJECT ACCESS
block.input["title"]              # ✓  block is SDK obj (dot), input is dict (brackets)
block.input.title                 # ✗  AttributeError
# Mental model: SDK metadata = dot. JSON payload = brackets.

# PRINTING SDK OBJECTS
print(f"x: {x}")                  # ✓  always works (uses __repr__)
json.dumps(response.content)      # ✗  TypeError on Pydantic blocks
[b.model_dump() for b in blocks]  # ✓  if you need real JSON

# NEXT() WITHOUT FALLBACK
next((b.text for b in content if b.type == "text"), "")  # ✓ default value
next(b.text for b in content if b.type == "text")        # ✗ StopIteration

# IS_ERROR SEMANTICS — DO NOT INVERT
return result, False              # success
return error_msg, True            # error
# Narrate aloud: "Returning success-data with is_error=False"

# KWARGS UNPACKING — SCHEMA PROPS MUST MATCH FUNCTION PARAMS
schema: {"properties": {"query": {"type": "string"}}}
fn(**block.input)                 # → fn(query="...")
def fn(query: str)                # ✓ matches
def fn(expr: str)                 # ✗ TypeError

# TUPLE vs DICT RETURNS FROM TOOLS
return [(start, end), ...]        # ✗ model guesses positions
return [{"start": s, "end": e}]   # ✓ named fields

# WHILE vs FOR(RANGE) FOR LOOPS
while response.stop_reason == "tool_use":  # ✗ no max_turns
for turn in range(max_turns):              # ✓ bounded

# STRING COMPARISON ON TIMES — only safe with zero-padding!
"11:00" < "2:00"   # → True  (because '1' < '2' lexicographically — WRONG)
# Use minutes-since-midnight: int(h)*60 + int(m), THEN compare.

Stdlib likely needed — memorize the call:

time.time()                            # → float seconds since epoch
from datetime import datetime, timedelta
datetime.now() - last_t                # → timedelta
hash((message, severity))              # → int (in-process dedup OK)
json.dumps(obj, default=str)           # → fallback for non-serializable
re.search(r"\d+", text)                # → match obj or None
§05Trace+Eval

See the loop. Score the loop.

Two helpers. Both fit on one screen. Both visible to the interviewer as you build.

Logging — one line per block

Print in two places only: after every messages.create(), and after every dispatch. Nothing else. The loop's change is fully described by what the model just said + what each tool returned. The messages array is cumulative state — don't dump it.

def log_turn(n, response):
    print(f"\n── turn {n} · stop={response.stop_reason} ──")
    for block in response.content:
        if block.type == "text":
            print(f"  text  · {block.text[:120]}")
        elif block.type == "thinking":
            print(f"  think · {block.thinking[:120]}")
        elif block.type == "tool_use":
            print(f"  →  {block.name}({block.input})")

def log_result(name, result, is_error):
    glyph = "✗" if is_error else "✓"
    print(f"  {glyph}  {name}: {str(result)[:120]}")

Wire-in (two lines in the loop):

for turn in range(max_turns):
    response = client.messages.create(...)
    log_turn(turn, response)                          # ← here
    if response.stop_reason == "end_turn": ...
    messages.append({"role": "assistant", "content": response.content})
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result, is_error = call_tool(block.name, block.input)
            log_result(block.name, result, is_error)     # ← and here
            tool_results.append({...})
    messages.append({"role": "user", "content": tool_results})

Why these rules:

  • One line per block — eyes scan vertically; glyph column lines up.
  • Truncate to ~120 chars — a full SQL query or row dump destroys the trace.
  • Glyphs over labels — → = call, ✓/✗ = result. Less text, more signal.
  • Always print stop_reason — single most important signal in the loop.
  • Don't print the full messages array — it grows every turn. Trace change, not state.

Eval — 4 cases, substring match

If 5 minutes remain, this is the bonus that lands. Run it before saying "done."

TESTS = [
    ("How many customers are from France?",       "2"),
    ("Total revenue from France-based customers?", "1575"),
    ("Which product has the most orders?",         "Laptop"),
    ("What is 25 + 30 * 2?",                       "85"),
]

def run_eval(tests):
    passed = 0
    for q, expected in tests:
        try:
            answer = run_agent(q)
            ok = expected.lower() in str(answer).lower()
        except Exception as e:
            answer, ok = f"ERROR: {e}", False
        glyph = "✓" if ok else "✗"
        print(f"{glyph} {q}")
        print(f"   expected: {expected}")
        print(f"   got:      {str(answer)[:120]}\n")
        passed += ok
    print(f"── {passed}/{len(tests)} passed ──")

Say aloud while it runs: "Substring match is a smoke test. Real eval is 5–10 verifiable prompts, log tool-call counts and tokens per case, LLM-as-judge calibrated on the first 20 — that's the Writing Tools for Agents guidance." One sentence. Then move on.

§06Reference

Quick reference, for when memory fails.

All stop_reason values to handle

"end_turn"                        → done; return text
"tool_use"                        → execute tools, append, continue
"max_tokens"                      → truncated; re-issue with higher max_tokens
"pause_turn"                      → server tool iteration cap; re-send as-is
"stop_sequence"                   → rare; treat as end_turn
"refusal"                         → safety; return refusal text
"model_context_window_exceeded"   → terminal; return error

tool_result block — required shape

{
    "type": "tool_result",
    "tool_use_id": block.id,   # MUST match the tool_use id
    "content": str(result),    # or list of {type:"text", text:...}
    "is_error": is_error,      # bool — wire through dispatcher
}

Wrong param typesAdd "strict": True or input_examples

disable_parallel_tool_use ignoredSet on the request that returns tool_use, not later

§07Opening

The 90-second opening move.

When the brief is done, this is the script. Walk through it; then code.

Got it. Before I start coding, let me read the starter end-to-end and tell you what I see — that'll also tell us what's missing.

[narrate aloud: client setup, MODEL, example tools, dispatcher pattern, loop shape, what stop_reasons it handles, what error handling exists]

Plan: I'll build [N] tools — [tool A], [tool B], maybe [tool C]. Read-only first, then mutating with preconditions. This is workflow-shaped enough that a single agentic loop is right — I wouldn't reach for routing or multi-agent here. One question before I start: [clarifying Q about the brief or the env].

Let me build the minimal end-to-end with [tool A] first, then iterate on the descriptions before adding the rest.

Ninety seconds. Hits all four non-Shipping axes simultaneously. Then code.
